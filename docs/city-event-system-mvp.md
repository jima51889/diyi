# 城市探秘动态事件系统 MVP

## 目标

把产品从“固定路线打卡”升级为“城市事件探索”。

V1 不推翻《雾都失落档案》主线，而是在主线之外增加一层可运营内容：

- 今日限时事件
- 随机任务包
- 玩家身份分配
- 神秘信封/商家线索占位
- 分享战绩素材

用户打开 H5 时看到的核心问题从“我要玩哪条路线”变成：

> 今天重庆又发生了什么？

## MVP 玩法结构

### 1. 固定主线

用于第一次付费验证。

当前主线：

- 城市：重庆
- 剧本：S1《雾都失落档案》
- 结构：五章
- 验证：GPS、照片、答题
- 结果：调查员身份卡

### 2. 今日事件

用于提升复访和分享。

当前示例：

- 事件：洪崖洞异常信号事件
- 类型：daily
- 稀有度：limited
- 有效期：Seed 时动态生成 3 小时倒计时
- 玩家领取后随机获得任务和身份

### 3. 随机任务包

同一个事件下可配置多个任务：

- PHOTO：拍摄城市现场
- QA：输入线索密码
- CLUE：神秘信封/商家线索占位

当前示例任务：

- 信号源确认
- 线人留言
- 神秘信封占位

### 4. 玩家身份

领取事件时随机分配身份：

- 调查员
- 破译师
- 追踪者
- 情报员

后续可以让不同身份拿到不同线索，形成玩家交换、组队和社交传播。

### 5. 神秘信封系统

MVP 先保留数据结构和 H5 展示占位。

未来可以接入：

- 合作商家
- 实体信封
- 线索卡
- 限定徽章
- 商家核销 QR

## 后端数据模型

新增核心表：

- `city_events`：城市事件
- `event_tasks`：事件任务池
- `player_roles`：玩家身份
- `user_event_assignments`：用户事件领取记录
- `clue_drops`：线索投放点/神秘信封点

## API

### 今日事件

`GET /api/events/today?city=重庆`

返回当前城市正在进行的事件列表。

### 当前事件领取记录

`GET /api/events/assignment?city=重庆`

需要登录。

### 领取事件

`POST /api/events/:id/claim`

需要登录。

领取后随机分配：

- 一个 `event_task`
- 一个 `player_role`

### 事件答题

`POST /api/events/tasks/:id/answer`

需要登录。

用于 QA 任务。

### 事件模拟通过

`POST /api/events/tasks/:id/mock-pass`

需要登录。

用于 PHOTO、CLUE、QR 等 MVP 阶段模拟完成。

## H5 页面

- 分享卡生成静态图

## NPC 闯关节点配置

主线任务中的真人互动先作为 `quest_nodes.node_type = challenge` 实现。

MVP 阶段配置存放在 `quest_nodes.reward` JSON 内，Admin 节点编辑页可配置：

- `npcRole`：NPC/线人角色
- `secretCode`：接头暗号
- `challengeRules`：闯关规则，一行一条
- `futureVerify`：核销方式，如 `staff_code`、`npc_qr`、`timed_challenge`、`mock_pass`
- `staffCode`：工作人员/NPC 核销码
- `commissionAmount`：NPC 服务佣金
- `objective`：玩家任务目标
- `shareTitle`：完成后的分享标题

开发模式下通过 `/api/game/challenge` 模拟通过；正式版可替换为 NPC 核销码、商家扫码、小程序小游戏结果或人工确认。

第二阶段再接入真实运营：

- 商家线索点
- QR 核销
- 实体信封
- 玩家交换线索
- 稀有事件和排行榜

第三阶段再做平台化：

- 创作者配置事件任务
- 平台审核
- 城市运营人员发布事件
- 商家/NPC 服务结算
